A Secret in Kubernetes is used to store sensitive data securely and provide it to containers when they run.

Sensitive data examples:
	• database passwords
	• API keys
	• tokens
	• SSH keys
	• TLS certificates

Instead of writing these values directly in a Pod YAML, Kubernetes stores them in a Secret object.

Why Secrets Are Needed
Imagine you deploy an application that connects to a database.

Without Secret:
env:
- name: DB_PASSWORD
  value: mypassword123
Problems:
	• password visible in YAML
	• stored in Git
	• security risk

With Secret:
env:
- name: DB_PASSWORD
  valueFrom:
    secretKeyRef:
      name: db-secret
      key: password

Password is stored separately and injected securely.

Secret Example
apiVersion: v1
kind: Secret
metadata:
  name: db-secret
type: Opaque
data:
  username: cm9vdA==
  password: cGFzc3dvcmQ=

Explanation:
Field	Meaning
apiVersion	Kubernetes API version
kind	Resource type
metadata.name	Secret name
type	Secret category
data	Base64 encoded values
	
Important:
Secrets use Base64 encoding, not encryption.

Example:
root → cm9vdA==
password → cGFzc3dvcmQ=

How Pods Use Secrets
Secrets can be injected into containers in three ways.

Method 1 — Environment Variables
Example :
env:
- name: DB_USER
  valueFrom:
    secretKeyRef:
      name: db-secret
      key: username

Inside container:
echo $DB_USER

Output:
root

Method 2 — Mount Secret as Files
Example:
volumes:
- name: secret-volume
  secret:
    secretName: db-secret
containers:
- name: app
  image: nginx
  volumeMounts:
  - name: secret-volume
    mountPath: /etc/secrets

Inside container:
/etc/secrets/username
/etc/secrets/password

Each key becomes a file.

Method 3 — Entire Secret as Environment Variables
envFrom:
- secretRef:
    name: db-secret

Container receives:
DB_USER=root
DB_PASSWORD=password

Internal Working of Secrets

Step 1 — Secret Creation
User runs:
kubectl apply -f secret.yaml
or
kubectl create secret generic db-secret \
  --from-literal=username=root \
  --from-literal=password=password

Step 2 — API Server Stores Secret
The Secret object is stored in:
etcd

Stored format:
Secret:
 name: db-secret
 data:
   username: cm9vdA==
   password: cGFzc3dvcmQ=

If encryption is enabled, data is encrypted before storing.

Step 3 — Pod References Secret
Example Pod spec:
env:
- name: DB_PASSWORD
  valueFrom:
    secretKeyRef:
      name: db-secret
      key: password

This means:
Pod needs Secret db-secret

Step 4 — Pod Scheduled to Node
The kube-scheduler assigns the Pod to a Node.

Example:
Pod → Node2

Step 5 — kubelet Fetches Secret
On Node2, kubelet:
	1. Reads Pod specification
	2. Sees Secret reference
	3. Requests Secret from API Server

Step 6 — kubelet Injects Secret
Depending on usage:

Environment variables
Kubelet injects variables into container runtime.

Example:
DB_PASSWORD=password

Volume mount
Kubelet creates files on the node:
/var/lib/kubelet/pods/<pod-id>/volumes/kubernetes.io~secret/

Files created:
username
Password

Then mounted inside container.

Types of Secrets
Kubernetes supports different types.

Type	Purpose
Opaque	generic secrets
kubernetes.io/dockerconfigjson	container registry login
kubernetes.io/tls	TLS certificates
kubernetes.io/service-account-token	service account tokens
	
Example TLS secret:
type: kubernetes.io/tls
Used for HTTPS with Ingress.

Security Notes
Secrets are:
	• stored in etcd
	• base64 encoded
	• optionally encrypted
Better security practices:
	• enable etcd encryption
	• restrict RBAC access
	• avoid printing secrets in logs
